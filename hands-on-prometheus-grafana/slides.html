Monitoring &
Observability

Mengapa kita perlu memantau server?

Apa itu Monitoring?

Monitoring adalah proses mengumpulkan, mencatat, dan menganalisis data dari sistem/layanan secara berkelanjutan untuk memastikan ketersediaan dan performa yang optimal.

Deteksi Masalah

Mengetahui ketika server atau layanan mengalami gangguan sebelum pengguna melaporkan

Kapasitas

Merencanakan kebutuhan sumber daya (CPU, RAM, disk) berdasarkan tren historis

Performa

Mengidentifikasi bottleneck dan mengoptimalkan respons waktu aplikasi

Keamanan

Mendeteksi pola akses anomali dan serangan potensial

Tiga Pilar Observability

Metrics

Data numerik berbasis waktu (time-series). Contoh: CPU usage 72%, request rate 150 req/s, latency P95 = 320ms

Prometheus

Logs

Catatan kejadian diskret yang terjadi di sistem. Contoh: error log, access log, audit trail

Loki / ELK

Traces

Jejak perjalanan request melewati berbagai layanan/microservice dari ujung ke ujung

Jaeger / Tempo
Hands-on ini berfokus pada pilar pertama: Metrics

Evolusi Monitoring

Generasi 1 (1988) Generasi 2 (2000-an) Generasi 3 (2012+) -------------------- ---------------------- ---------------------- SNMP Nagios / Zabbix Prometheus / Grafana (Pull/Push OID) (Check-based, agent) (Pull metrics, TSDB) Network devices Server infra Cloud-native / k8s Sederhana Lebih kaya fitur Skalabel, label-based MIB-based Threshold alerts Flexible PromQL UDP port 161 Active checks Scrape model

Ketiga generasi masih relevan dan digunakan bersama di lingkungan enterprise

02

SNMP

Simple Network Management Protocol

Apa itu SNMP?

SNMP adalah protokol standar Internet (RFC 1157, 1993) untuk mengumpulkan dan mengatur informasi perangkat jaringan: router, switch, server, printer, UPS.

Karakteristik

  • 1Berjalan di atas UDP port 161 (queries) dan 162 (traps)
  • 2Mendukung GET, SET, TRAP, GetNext, GetBulk
  • 3Data diorganisir dalam hierarki MIB
  • 4Tiga versi: v1, v2c, v3 (paling aman)

Digunakan Untuk

  • Monitor bandwidth network
  • CPU & memory perangkat
  • Status interface jaringan
  • Deteksi link down (trap)

Komponen SNMP

┌─────────────────────────────────────────────────────────┐ │ SNMP Architecture │ │ │ │ ┌──────────────────┐ ┌───────────────────────┐ │ │ │ NMS / Manager │ │ Managed Device │ │ │ │ (Zabbix, PRTG, │──────▶│ (Router, Switch, │ │ │ │ LibreNMS) │ GET │ Server, Printer) │ │ │ │ │◀──────│ │ │ │ │ Mengumpulkan, │ Response ┌───────────────┐ │ │ │ │ menampilkan, │ │ │ SNMP Agent │ │ │ │ │ memberi alert │◀──────│ │ (daemon) │ │ │ │ └──────────────────┘ TRAP │ └───────────────┘ │ │ │ ▲ │ ┌───────────────┐ │ │ │ │ │ │ MIB Objects │ │ │ │ UDP 162 │ │ (OID tree) │ │ │ │ │ └───────────────┘ │ │ │ └───────────────────────┘ │ └─────────────────────────────────────────────────────────┘

MIB & OID

MIB (Management Information Base) adalah kamus yang mendefinisikan semua objek yang dapat di-query pada perangkat. Setiap objek memiliki OID (Object Identifier) berupa angka bertitik. 
# Contoh OID penting (MIB-II / RFC 1213)
.1.3.6.1.2.1.1.1.0   # sysDescr  (deskripsi sistem)
.1.3.6.1.2.1.1.5.0   # sysName   (hostname)
.1.3.6.1.2.1.1.3.0   # sysUpTime

# CPU load (UCD-SNMP-MIB)
.1.3.6.1.4.1.2021.11.9.0   # load 1min

# Interface in/out octets
.1.3.6.1.2.1.2.2.1.10.1   # ifInOctets  eth0
.1.3.6.1.2.1.2.2.1.16.1   # ifOutOctets eth0

SNMP Commands

# Query (GET)
snmpget -v2c -c public \
  192.168.1.1 \
  .1.3.6.1.2.1.1.1.0

# Walk (semua OID)
snmpwalk -v2c -c public \
  192.168.1.1 system

# Versi 3 (lebih aman)
snmpget -v3 -u admin \
  -A mypassword \
  -l authPriv \
  192.168.1.1 sysUpTime.0

FiturSNMPv1SNMPv2cSNMPv3
Tahun198819931998
Autentikasi Community string (plaintext) Community string (plaintext) Username + Hash (MD5/SHA)
Enkripsi Tidak ada Tidak ada DES / AES-128/256
GetBulkTidakYaYa

Keterbatasan SNMP

Pull-only (default)

Manager harus aktif polling ke setiap device; tidak efisien untuk ribuan node

MIB Complexity

Vendor sering proprietary MIB; sulit dikompilasi dan dikelola

Tidak Cloud-Native

Container, microservice, dan ephemeral service tidak cocok dengan model SNMP

Tidak Flexible

Sulit melakukan query ad-hoc atau agregasi kompleks dibanding PromQL

Inilah sebabnya lahir tools generasi baru seperti Prometheus
03

Prometheus

Open-source Monitoring & Alerting Toolkit

Apa itu Prometheus?

Prometheus adalah sistem monitoring time-series open-source yang dikembangkan di SoundCloud (2012) dan sekarang dikelola oleh CNCF (Cloud Native Computing Foundation). Menjadi standar de-facto untuk monitoring Kubernetes.
Pull Model Prometheus yang mendatangi target untuk mengambil metrics (scrape)
TSDB Time Series Database bawaan; efisien untuk data metrics
PromQL Query language yang powerful untuk analisis metrics
Labels Dimensi fleksibel; satu metric bisa punya banyak label 
# prometheus.yml
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: "webserver1"
    static_configs:
      - targets:
          - "webserver1:5000"

  - job_name: "webserver2"
    static_configs:
      - targets:
          - "webserver2:5000"

Tipe Metrics Prometheus

Counter

Nilai yang hanya naik (tidak pernah turun). Reset ke 0 saat restart.

http_requests_total{method="GET", status="200"} 1024
Selalu naik
Gauge

Nilai yang bisa naik atau turun setiap saat.

cpu_usage_percent 72.4 | requests_in_progress 5
Naik/turun
Histogram

Distribusi nilai ke dalam bucket; mendukung percentile (P50, P95, P99).

http_request_duration_seconds_bucket{le="0.1"} 872
Percentile
Summary

Serupa histogram tapi quantile dihitung di sisi client; tidak bisa diagregasi.

rpc_duration_seconds{quantile="0.95"} 0.321
Client-side

Instrumentasi dengan Python

Library prometheus-client mempermudah instrumentasi aplikasi Python/Flask. 
from flask import Flask, Response, request
from prometheus_client import Counter, Histogram, Gauge
from prometheus_client import generate_latest, CONTENT_TYPE_LATEST
import time

app = Flask(__name__)

# Definisikan metrics
REQUEST_COUNT = Counter(
    'http_requests_total', 'Total HTTP requests',
    ['server', 'method', 'endpoint', 'http_status']
)
REQUEST_LATENCY = Histogram(
    'http_request_duration_seconds', 'Request latency',
    ['server', 'method', 'endpoint']
)

@app.before_request
def start_timer():
    request._start_time = time.time()

@app.after_request
def record_metrics(response):
    duration = time.time() - request._start_time
    REQUEST_COUNT.labels('webserver1', request.method,
                         request.path, response.status_code).inc()
    REQUEST_LATENCY.labels('webserver1', request.method,
                            request.path).observe(duration)
    return response

@app.route('/metrics')
def metrics():
    return Response(generate_latest(), mimetype=CONTENT_TYPE_LATEST)

PromQL — Query Language

PromQL adalah bahasa query functional untuk menganalisis time-series data di Prometheus.
Query PromQLArti
http_requests_total Semua nilai counter saat ini
http_requests_total{server="webserver1"} Filter berdasarkan label
rate(http_requests_total[1m]) Laju request per detik (1 menit terakhir)
sum by (server) (rate(http_requests_total[1m])) Total req/s per server
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m])) P95 latency
rate(http_requests_total{http_status=~"5.."}[1m]) / rate(http_requests_total[1m]) Error rate (5xx / total)

Grafana

Open-source Analytics & Monitoring Platform

Apa itu Grafana?

Grafana adalah platform visualisasi open-source yang dapat terhubung ke berbagai data source (Prometheus, InfluxDB, MySQL, Elasticsearch, dll.) dan menampilkan data dalam dashboard yang interaktif.
Dashboard Kumpulan panel visualisasi yang bisa di-share
Panels Time series, Gauge, Bar chart, Stat, Table, dll.
Alerting Alert berbasis query, notifikasi ke Slack/Email/PagerDuty
Provisioning Konfigurasi as-code (YAML & JSON)
# Auto-provisioning datasource
apiVersion: 1

datasources:
  - name: Prometheus
    type: prometheus
    url: http://prometheus:9090
    isDefault: true

# Dashboard auto-load dari JSON
providers:
  - name: default
    type: file
    options:
      path: /etc/grafana/provisioning/dashboards

Tipe Panel Grafana

Time Series

Grafik garis untuk melihat tren nilai dari waktu ke waktu. Paling umum digunakan.

Gauge

Jarum penunjuk (speedometer), cocok untuk menampilkan nilai saat ini vs threshold.

Stat

Satu angka besar dengan warna. Cocok untuk total request, uptime, dll.

Bar Chart

Perbandingan nilai antar kategori/instance dalam bentuk batang.

Table

Tampilkan data dalam format tabel; bisa dikombinasikan dengan transformasi.

Heatmap

Visualisasi distribusi histogram; sangat berguna untuk melihat latency distribution.

PromQL &
Alerting Rule

Analisis Mendalam & Konfigurasi Alert

PromQL Lanjutan

# ─── Aggregasi ────────────────────────────────────────────────────────────
# Total request semua server, dikelompokkan per server
sum by (server) (http_requests_total)

# Rata-rata request rate per server (5 menit)
avg by (server) (rate(http_requests_total[5m]))

# ─── Filtering dengan regex ───────────────────────────────────────────────
# Hanya endpoint yang mengandung "api"
rate(http_requests_total{endpoint=~"/api/.*"}[1m])

# Kecualikan endpoint /metrics
rate(http_requests_total{endpoint!="/metrics"}[1m])

# ─── Kalkulasi Error Rate dalam persen ────────────────────────────────────
(
  sum by (server) (rate(http_requests_total{http_status=~"5.."}[2m]))
  /
  sum by (server) (rate(http_requests_total[2m]))
) * 100

# ─── Latency average per endpoint ─────────────────────────────────────────
sum by (server, endpoint) (rate(http_request_duration_seconds_sum[1m]))
/
sum by (server, endpoint) (rate(http_request_duration_seconds_count[1m]))

Alert Rules

# prometheus/alert_rules.yml
groups:
  - name: webserver_alerts
    rules:

      # Server tidak dapat diakses > 30 detik
      - alert: WebServerDown
        expr: up{job=~"webserver.*"} == 0
        for: 30s
        labels:
          severity: critical
        annotations:
          summary: "Web server {{ $labels.job }} is DOWN"

      # Error rate melebihi 10%
      - alert: HighErrorRate
        expr: |
          rate(http_requests_total{http_status=~"5.."}[2m])
          / rate(http_requests_total[2m]) > 0.1
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "High error rate on {{ $labels.server }}"

      # Latency P95 melebihi 2 detik
      - alert: HighLatency
        expr: |
          histogram_quantile(0.95,
            rate(http_request_duration_seconds_bucket[5m])
          ) > 2
        for: 2m
        labels:
          severity: warning

Perbandingan SNMP vs Prometheus

AspekSNMPPrometheus
Model Pull (GET) + Push (TRAP) Pull (Scrape)
Protokol UDP 161/162 HTTP /metrics
Format Data OID → Value metric{labels} value
Query OID tree walk PromQL (powerful)
Target Network devices Aplikasi, container, OS
Cloud-Native Tidak Ya (k8s ready)
Skalabilitas Terbatas Horizontal (federation)
Ekosistem Zabbix, PRTG, LibreNMS Grafana, Alertmanager

Ringkasan

SNMP

Protocol standar untuk monitoring network device. Masih relevan untuk router & switch. SNMPv3 untuk keamanan.

Prometheus

Scrape /metrics dari aplikasi, simpan di TSDB, query dengan PromQL. Pull-based, cloud-native.

Grafana

Dashboard visualisasi yang terhubung ke Prometheus. Auto-provisioning dengan YAML & JSON.

Docker

Orkestrasi semua komponen dalam satu network. Mudah dijalankan dengan docker compose up.

Alur data: Aplikasi → expose /metrics → Prometheus scrape → simpan TSDB → Grafana query → visualisasi dashboard
